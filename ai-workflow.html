01 — Toolchain

Speckit thinks. GSD builds.
I review.

My development stack is built around Claude Code with two plugins that handle different phases of work:

Speckit
Plan & specify
Spec-driven development. Define principles → specify requirements → generate implementation plan → break into dependency-ordered tasks. The AI does the thinking before it touches code.
GSD
Execute & verify
Fresh Claude instances per task. No context rot. Plan → execute → verify with atomic commits and built-in verification. For quick fixes: /gsd:quick skips planning entirely.
Me
Judge & correct
Review architecture decisions, catch security gaps, fix edge cases, and know when to skip AI entirely. The most important tool in the chain isn't a tool — it's judgment.

The decision tree is simple:

  • → Complex feature or new system? Speckit first (plan), then GSD per task (execute). Full spec-driven flow.
  • → Single well-scoped task? Straight to /gsd:quick "description". Same quality guarantees, faster path.
  • → Quick fix I already know how to solve? No AI. I open the file and fix it myself. Wasting tokens on a one-line change you already understand is an anti-pattern.
02 — Real Example

Building a fulfillment backend from scratch.

I recently built a complete fulfillment backend — order management, 3PL integration, payment processing, multi-tenant credential management — from zero to processing test orders in under three weeks. Here's how the workflow played out.

Phase 1: Speckit Planning

# Define system boundary, user stories, constraints
/speckit.specify → requirements + acceptance criteria

# Generate technical implementation plan
/speckit.plan → architecture, API design, data model

# Break into dependency-ordered tasks
/speckit.tasks → 7 core tables, API routes, integrations
Where my review mattered

The AI's first plan had the payment flow calling the shipping API synchronously inside the order creation endpoint. I caught this — payment confirmation and label generation are separate concerns that should be decoupled. Pushed it to an event-driven pattern with status tracking.

The AI doesn't understand your production failure modes. You do.

Phase 2: GSD Execution

# Execute each task group through GSD
/gsd:plan-phase 3 → ShipStation 3PL integration
/gsd:execute-phase 3 → webhook handler, label gen, tracking sync
/gsd:verify-work 3 → automated verification checks
Two catches during review

Security: The generated code stored API credentials in a config file. I redirected to AWS KMS encryption with per-tenant isolation — the AI defaulted to the simplest approach, not the secure one.

Reliability: Rate-limiting logic didn't account for the API's burst limits during batch operations. I restructured with queuing and exponential backoff based on actual API documentation the AI hadn't internalized.

Phase 3: Straight-to-Code Fixes

No AI needed

After integration testing, I found a PostgreSQL query returning stale inventory counts — missing join condition. I didn't spin up Claude for this. I already saw the issue in the query plan, opened the file, fixed the WHERE clause. Three-second fix. Using AI here would have been slower than just typing.

03 — Mental Model

Where AI excels vs. where it needs me.

After six months of daily production use, I've developed a clear framework:

✦ AI excels at

  • Scaffolding — schemas, boilerplate, CRUD, test structures
  • Pattern replication — once shown one integration, it replicates cleanly
  • Documentation and type definitions — tedious but reliable
  • Exploring unfamiliar APIs — faster than reading docs start to finish

✦ Needs human judgment

  • Architecture — data flow, service boundaries, coupling decisions
  • Security — credential storage, tenant isolation, encryption scope
  • Performance at scale — the AI writes code that works; I make it perform
  • Error handling in distributed systems — retry logic, partial failures, idempotency
  • Domain-specific edge cases — business logic the AI can't infer from code
04 — The Point

The skill isn't typing faster.
It's knowing when.

The value I bring isn't speed — it's judgment. Knowing when to delegate to the AI, when to take the wheel, and when to skip AI entirely is the actual skill. My Speckit + GSD workflow gives me structure, fresh contexts prevent quality drift, and the clear phase separation means I'm reviewing at every stage — not rubber-stamping output.

I've shipped a multi-agent AI platform (LangGraph, 40+ MCP tools, dual-layer RAG) and a production fulfillment backend this way. Both from scratch, both in months, both with measurable results.

The same approach — structured planning, AI-powered execution, human-driven review — is exactly how I'd build solutions for customers.
